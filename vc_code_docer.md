В «Школе 21» mini-verter — это ваш лучший друг перед сдачей проекта. Это локальный скрипт-тестер, который имитирует проверку основного робота (автотеста), чтобы вы не получили 0 за проект из-за какой-нибудь глупой ошибки в Makefile или структуре файлов.

Cвязка для «Школы 21»:
Docker Desktop (программа на Windows) — сам «двигатель».
Docker Extension (это расширение) — панель управления и мониторинга.
Dev Containers — инструмент для превращения вашей папки с кодом в полноценную Linux-среду внутри Docker.

Для запуска такого тестера Docker и Dev Containers — это идеальное решение, потому что робот Вертер работает на Linux (Ubuntu/Alpine), а вы сейчас на Windows.
Как правильно «подружить» проект и mini-verter в VS Code:

Если тестер требует Docker:

Вам нужно создать в папке проекта файл Dockerfile.

В нём обычно прописывается установка компиляторов (gcc, make) и самого тестера.

После этого расширение Dev Containers позволит вам «войти» в это окружение.

Как изменится ваш Makefile для mini-verter:

На Windows мы использовали cl.exe, но Вертер и mini-verter — это про Linux. Ваш Makefile для сдачи в Школу 21 должен выглядеть примерно так (стандартный шаблон):

makefile

    CC = gcc
    CFLAGS = -Wall -Wextra -Werror -std=c11
    SOURCES = main.c
    OBJECTS = $(SOURCES:.c=.o)
    TARGET = s21_program
    
    all: $(TARGET)
    
    $(TARGET): $(OBJECTS)
    	$(CC) $(CFLAGS) $(OBJECTS) -o $(TARGET)
    
    %.o: %.c
    	$(CC) $(CFLAGS) -c $< -o $@
    
    clean:
    	rm -rf $(OBJECTS) $(TARGET)
    
    rebuild: clean all

Почему это важно для mini-verter:

Флаги -Wall -Wextra -Werror: Без них mini-verter может даже не начать тесты. Это требование «Школы 21» к чистоте кода.

Команда rm -rf: На Linux (внутри Docker) для очистки используется именно она, а не del.

Что делать дальше:

Убедитесь, что Docker Desktop запущен у вас в системе.

Если у вас есть папка с mini-verter, положите её в корень вашего проекта.

Обычно запуск тестера в терминале (внутри контейнера) выглядит так: sh mini_verter.sh или через make test.

Вы уже скачали файлы mini-verter? Если да, посмотрите, нет ли там инструкции, какой именно Docker-образ он требует (обычно это указано в README тестера). Это поможет нам настроить Dev Containers так, чтобы тесты проходили с первого раза.

Что такое Dev Containers?

Представьте, что вы хотите программировать в среде Linux (как в школе), но сидите на Windows. Это расширение позволяет VS Code «нырнуть» внутрь изолированного контейнера (Docker), где установлены все нужные инструменты.

Если вам нужно собрать и запустить mini-verter (или похожий проект) в изолированной среде, тогда Dev Containers и Docker — это именно то, что доктор прописал! Это очень крутой навык для "Школы 21", так как там часто проекты проверяются в специфических Linux-окружениях.
Однако, чтобы эта «ракета» взлетела, нужно убедиться, что фундамент готов.

Что нужно, чтобы Dev Containers заработал:

-Docker Desktop: Он должен быть установлен в Windows и запущен (в трее должен быть китенок). Без него расширение в VS Code — это просто пустая кнопка - расширение Dev Containers будет выдавать ошибку при попытке запуска.

-WSL 2: Docker на Windows лучше всего работает через подсистему Linux (Windows Subsystem for Linux).

-Папка .devcontainer: В вашем проекте должна появиться папка с этим названием, а в ней файл devcontainer.json. В нем описывается, какой образ Linux использовать и какие расширения (типа C++ и Makefile) установить внутри контейнера.

Как это меняет работу с Makefile:

Когда вы работаете через Dev Containers: Ваш компилятор cl.exe (Windows) больше не нужен. Внутри контейнера будет работать стандартный линуксовый gcc или clang.

Makefile станет "настоящим" (линуксовым). Вам не нужно будет писать del для очистки, вы будете писать rm -f.
Расширение Makefile Tools будет работать прямо внутри Docker, подхватывая все зависимости.

Ваш следующий шаг для веб-сервера:
Если Docker Desktop уже установлен:

Нажмите на синий значок >< в левом нижнем углу VS Code.

Выберите "Add Dev Container Configuration Files...".

Выберите подходящий образ (например, C++).

VS Code предложит "Reopen in Container". Нажмите это, и вы окажетесь внутри Linux, не выходя из Windows.

Docker от Microsoft — это основной «пульт управления», а Dev Containers (который вы смотрели до этого) — это его дополнение. Они работают в паре.

Для чего оно нужно в дополнение к Dev Containers:

Визуальный контроль: В левой панели появится иконка «кита». Нажав на неё, вы увидите все свои запущенные контейнеры, образы и сети. Это гораздо удобнее, чем постоянно писать команды в терминале.

Управление файлами: Вы сможете кликнуть правой кнопкой мыши по запущенному контейнеру и выбрать «Attach Shell», чтобы мгновенно попасть внутрь Linux-среды.

Логи: Если ваш mini-verter или веб-сервер не запускается, через это расширение можно в один клик посмотреть логи ошибок.

Вердикт: Если вы планируете использовать mini-verter, то это расширение — обязательный компаньон. 

Минусы:
Сложность: Чтобы оно работало, на ваш компьютер нужно установить огромную и тяжелую программу — Docker Desktop. Без нее это расширение просто бесполезно.

Путаница: Вам придется разбираться не только в языке Си, но и в том, как настраивать контейнеры, пробрасывать в них папки и настраивать сеть.

Ресурсы: Docker потребляет много оперативной памяти.
Вердикт: Если вы не планируете прямо завтра разворачивать серверы или работать в изолированных Linux-средах на Windows, это расширение можно смело удалять или отключать.
